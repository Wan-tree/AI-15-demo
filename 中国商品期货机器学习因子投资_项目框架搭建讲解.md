# 中国商品期货机器学习因子项目：从零搭建讲解文档

## 一、项目总体结论

这个项目可以先搭建成一个“不依赖机器学习、但从数据到回测全流程可运行”的研究框架。

第一阶段的正确目标不是训练复杂模型，而是打通下面这条链路：

```mermaid
flowchart LR
    A["上财平台导出原始数据"] --> B["原始文件只读保存"]
    B --> C["字段标准化与质量检查"]
    C --> D["期货合约与主力换月处理"]
    D --> E["基础因子与预测标签"]
    E --> F["信号与组合构建"]
    F --> G["含成本的历史回测"]
    G --> H["指标、图表和研究报告"]
    E -.未来扩展.-> I["机器学习模型"]
    I --> F
    H -.未来扩展.-> J["LLM 研究助手"]
```

最终框架需要做到：

- 没有上财数据时，使用项目自带的合成示例数据完整运行；
- 拿到上财导出的 CSV/Excel 后，只需要配置字段映射，不重写研究代码；
- 以后可以插入 LightGBM、XGBoost、神经网络或 LLM；
- 数据处理、因子、组合、回测彼此分离，方便多人协作；
- 每一步都有数据检查、日志、配置和测试；
- 不把学校数据库原始数据、账号和密码上传到 Git。

## 二、上财实验数据资源平台怎么使用

访问入口：[上财实验数据资源平台](https://db.sufe.edu.cn/reservation/?tenant=shufelims)

上财实验中心公开说明显示，这并不是一个单纯的“网页下载数据库”，而是一站式实验资源平台。资源包括：

- 虚拟应用数据库镜像，如 Wind、Choice、同花顺、恒生聚源等；
- 虚拟桌面；
- 在线访问数据库，如 CSMAR、RESSET 等；
- 线下数据库终端；
- 落地 SQL 数据库；
- Anaconda、Matlab、Stata、RStudio 等实验软件。

其中虚拟桌面还明确包含“商品期货交易理论与实务”“量化投资”“程序化交易与实现”等实验环境。参考：[上财实验中心官方说明](https://lab.sufe.edu.cn/sysjzy_11869/list.htm)。

### 2.1 推荐操作顺序

#### 第一步：准备浏览器和插件

官方推荐使用较新版本的 Chrome 或 Firefox。

如果准备使用“直连登录”：

1. 在网站上下载与你操作系统匹配的虚拟镜像专用插件；
2. 安装插件；
3. 直连预约成功后，网站会下载一个 `.ica` 文件；
4. 双击 `.ica` 文件进入远程桌面。

`.ica` 通常由 Citrix 类客户端打开，但应优先安装网站提供或指定的版本，不要随意从第三方网站下载。

如果使用“网页访问”，可能不必处理 `.ica` 文件，但需要允许浏览器弹窗。

#### 第二步：登录

1. 打开上述网址；
2. 点击右上角“登录”；
3. 使用学校允许的统一身份认证方式登录；
4. 允许该网站打开弹窗；
5. 登录后刷新一次页面。

官方特别说明：不先登录，镜像可能无法正常使用；网页访问时若禁止弹窗，也可能无法打开。

#### 第三步：查找适合商品期货的数据资源

可以使用：

- 右上角“快速导航”选择镜像类别；
- 浏览器 `Ctrl + F` 搜索 `Wind`、`Choice`、`同花顺`、`CSMAR`、`RESSET`、`商品期货`、`期货` 或 `金融数据分析`。

不要一开始同时研究所有数据库。建议先确认一个数据源能否提供以下数据：

- 中国商品期货所有历史合约日行情；
- 合约代码、品种代码和交易所；
- 开高低收、结算价、成交量、持仓量；
- 合约上市日和最后交易日；
- 合约乘数、最小变动价位；
- 涨跌停、交易日历；
- 手续费或可用于估计交易成本的信息。

#### 第四步：进入数据库

平台通常有两种方式：

- **直连登录**：下载 `.ica` 文件，通过客户端进入镜像；
- **网页访问**：直接在浏览器中进入远程环境。

第一次建议尝试“网页访问”；如果浏览器方式卡顿、下载受限或文件无法保存，再使用“直连登录”。

#### 第五步：在数据库内部查询

进入 Wind、Choice、CSMAR 或 RESSET 后，需要在数据库自己的界面中查询。不同数据库操作差别很大，但查询目标应保持一致：

1. 市场范围：国内商品期货，不包括股指、国债等金融期货；
2. 数据频率：先选择日频；
3. 合约范围：下载全部具体月份合约，不要只下载数据库提供的主力连续指数；
4. 时间范围：先用少量品种和一年数据测试，确认正确后再扩大范围；
5. 输出格式：优先 CSV，其次 XLSX；
6. 编码：优先 UTF-8；
7. 每次导出都保存字段说明和查询条件。

上财图书馆提供了 RESSET 用户手册、高频系统手册和数据字典，但本项目第一版不建议从高频数据开始。参考：[上财图书馆数据库利用指南](https://lib.sufe.edu.cn/sjklyzn/list.htm)。

### 2.2 预约限制

官方公开说明中有几个重要限制：

- 最多可以同时预约三个不同镜像；
- 预约满后，需要在“查看预约”中撤回或删除；
- 快速预约通常覆盖连续三个节次；
- 到达最后一个节次后，可再次预约延长；
- 应在预约规定时间内使用。

所以第一次不要同时预约很多资源，先选择一个最容易导出数据的数据库。

### 2.3 校外访问

上财图书馆说明，校外访问方式可能包括：

- 学校 VPN；
- CARSI/Shibboleth；
- 特定数据库账号。

但 Wind、Bloomberg、聚源、同花顺等资源可能只允许指定终端访问。因此，能否直接从校外进入某个数据库，要以该资源页面的具体说明为准。参考：[上财图书馆校外访问说明](https://lib.sufe.edu.cn/8324/list.htm)。

### 2.4 当前项目对上财网站的合理定位

从当前公开说明看，没有发现该预约平台面向个人研究项目提供公开 REST API 的明确文档。因此第一版不应该写“自动爬取上财网站”的程序。

推荐的数据入口是：

> 在上财授权环境内人工查询、导出文件 → 将文件复制到本地项目 → 项目自动读取、检查和处理。

如果以后确认某个数据库提供官方 Python API、SQL Server 连接、数据终端插件或授权数据接口，再增加对应的数据连接器。不要抓取网页、保存登录 Cookie 或在代码里写学校账号密码。

## 三、第一版项目研究目标

建议将研究问题定义为：

> 使用中国商品期货日频合约数据，构造不存在未来信息的连续研究序列和基础量价因子，进行横截面排序、组合构建和历史回测，为后续机器学习预测收益或排序模型提供统一数据集和接口。

第一版暂不实施：

- 实盘交易；
- 分钟或逐笔高频数据；
- 深度学习；
- LLM 自动交易；
- 自动登录学校数据库；
- 复杂分布式计算；
- 期权或跨境期货。

先完成日频商品期货研究，已经足够形成一个完整、可信的课程项目。

## 四、第一版应该实现的功能

### 4.1 数据接入

支持读取：

- CSV；
- XLSX；
- Parquet；
- 项目自带的合成演示数据。

不同数据库导出的字段名可能不同，例如：

| 标准字段 | 数据库里可能出现的名称 |
|---|---|
| `trade_date` | 交易日期、日期、TradingDate |
| `contract` | 合约代码、证券代码、WindCode |
| `close` | 收盘价、Close |
| `settlement` | 结算价、SettlementPrice |
| `volume` | 成交量、Volume |
| `open_interest` | 持仓量、OI、OpenInterest |

字段映射应写在配置文件中，不能散落在程序代码里。

### 4.2 数据质量检查

程序需要自动检查：

- 日期是否能正确解析；
- 主键“交易日 + 合约”是否重复；
- 价格是否为负数或零；
- `high` 是否小于 `low`；
- 成交量、持仓量是否为负；
- 合约代码能否解析出品种和月份；
- 日期是否超出合约上市/到期范围；
- 数据是否出现大段缺失；
- 是否缺少关键字段；
- 同一字段单位是否一致。

严重错误应停止运行，而不是默默跳过。

### 4.3 期货合约处理

这是项目里最重要、也最容易出错的部分。

必须保留所有原始月份合约，例如 `RB2610`、`CU2609`、`M2701`，然后每天根据前一交易日的持仓量或成交量选择主力合约。

推荐规则：

1. 使用前一交易日持仓量选择当日交易合约，避免使用当天收盘后才知道的信息；
2. 不允许主力合约频繁倒退到更早到期月份；
3. 临近最后交易日时强制换月；
4. 保存每天“品种 → 实际合约”的映射；
5. 技术指标可使用复权连续序列；
6. 回测盈亏必须使用真实合约价格和换月记录。

不能直接把数据库的“主力连续价格”当成真实可交易价格。

### 4.4 因子框架

第一版可以实现少量经典基础因子：

- 时间序列动量；
- 短期反转；
- 波动率；
- 成交量变化；
- 持仓量变化；
- 流动性；
- 期限结构因子——前提是数据包含不同月份合约；
- 价格相对移动平均线偏离程度。

每个因子都统一接受标准化行情表，并输出：

```text
trade_date | product | factor_name | factor_value
```

### 4.5 标签生成

为未来机器学习提前准备：

- 下一交易日收益；
- 未来 5 日收益；
- 未来 20 日收益；
- 横截面收益排名；
- 收益方向标签。

第一版只生成这些标签，不训练复杂模型。

### 4.6 信号和组合

基本流程：

1. 每天对各商品因子值进行横截面排序；
2. 做多排名靠前的品种；
3. 做空排名靠后的品种；
4. 对品种进行等权或波动率倒数加权；
5. 保持多空名义金额大致平衡；
6. 限制单品种权重；
7. 根据流动性过滤不可交易品种。

### 4.7 回测

回测至少需要考虑：

- 信号生成时间和下单时间；
- 合约乘数；
- 价格变动产生的真实盈亏；
- 手续费；
- 滑点；
- 换月成本；
- 涨跌停时无法成交；
- 缺失行情；
- 保证金占用的简化处理；
- 每日持仓和换手率。

第一版可以是日频向量化回测，不必搭建复杂事件驱动交易系统。

### 4.8 结果评价

输出：

- 年化收益率；
- 年化波动率；
- Sharpe；
- 最大回撤；
- Calmar；
- 胜率；
- 换手率；
- 多头、空头贡献；
- 各品种收益贡献；
- 因子 IC 和 Rank IC；
- 分组收益；
- 净值曲线；
- 回撤曲线；
- 月度收益表。

### 5.1 关键文件解释

- `README.md`：项目入口，告诉同伴如何安装、运行和放置数据；
- `pyproject.toml`：项目名称、Python 版本、依赖和开发工具配置；
- `.gitignore`：禁止提交原始数据、报告缓存、密码、虚拟环境；
- `.env.example`：只展示未来 API 配置名称，不放真实密码；
- `configs/`：放所有可调整参数，避免修改代码；
- `data/raw/`：上财平台原始导出文件，永远不覆盖；
- `data/interim/`：字段已标准化但尚未完成期货处理的数据；
- `data/processed/`：连续合约、因子、标签等研究数据；
- `data/manifests/`：记录来源、导出时间、查询条件、文件校验值；
- `docs/`：数据说明、方法说明和新手术语；
- `notebooks/`：用于探索和展示，不承担核心生产逻辑；
- `src/`：真正可复用、可测试的程序；
- `tests/`：保证换月、收益计算和因子不存在明显错误；
- `artifacts/`：回测图表和报告，通常不提交 Git。

## 八、未来真正搭建时的安装方式

推荐采用 Conda 环境加项目本身的依赖管理：

```bash
conda create -n cn-futures-factor python=3.11 -y
conda activate cn-futures-factor

python -m pip install --upgrade pip
pip install -e ".[dev]"
```

其中：

- `cn-futures-factor` 是独立环境名；
- `-e` 表示以可编辑方式安装项目；
- `[dev]` 同时安装测试和代码质量工具；
- 以后修改 `src/` 中的代码，不必重复安装。

检查环境：

```bash
python --version
python -m cn_futures_factor doctor
pytest
```

不建议直接在系统 Python 中安装所有库。

## 九、从零开始的实际搭建顺序

### 阶段 0：冻结研究口径

先写清楚：

- 研究市场：国内商品期货；
- 数据频率：日频；
- 是否包括夜盘；
- 使用收盘价还是结算价；
- 预测未来几日收益；
- 组合是横截面多空还是单品种择时；
- 回测区间；
- 成本假设。

### 阶段 1：调查上财数据

1. 登录平台；
2. 找到一个候选数据库；
3. 只导出一个品种、一个月或一年；
4. 保存字段说明；
5. 检查数据格式、代码格式和单位；
6. 确认能否导出所有月份合约；
7. 确认能否获得合约基本信息；
8. 再决定正式的数据源。

这一步不要直接下载全市场几十 GB 数据。

### 阶段 2：建立可运行空框架

创建项目目录、Python 包、配置系统、日志、命令行、合成示例数据和最小测试。

此时应该能运行：

```bash
python -m cn_futures_factor doctor
python -m cn_futures_factor demo
```

`demo` 应自动生成合成期货数据并输出一份简单回测报告。

### 阶段 3：接入上财导出文件

预计运行方式：

```bash
python -m cn_futures_factor ingest \
  --source sufe_export \
  --input data/raw/my_export.xlsx
```

程序完成读取文件、字段重命名、类型转换、合约代码解析、标准 Parquet 输出和质量检查报告。

### 阶段 4：处理期货合约

```bash
python -m cn_futures_factor prepare
```

完成品种和合约识别、主力合约选择、换月映射、连续研究序列、可交易收益及流动性过滤。

### 阶段 5：计算因子和标签

```bash
python -m cn_futures_factor research \
  --config configs/factors.yaml
```

输出因子值、未来收益标签、IC 分析、分组收益、缺失率和因子相关性。

### 阶段 6：基础回测

```bash
python -m cn_futures_factor backtest \
  --config configs/backtest.yaml
```

生成每日持仓、每日交易、手续费和滑点、换月记录、净值、风险指标及 HTML 或 Markdown 报告。

### 阶段 7：测试和交付

最终至少通过：

```bash
pytest
ruff check .
ruff format --check .
```

然后由另一位同伴在全新环境里按照 README 重新安装和运行一次。

## 十、商品期货项目最容易踩的坑

### 10.1 只下载主力连续行情

主力连续数据适合画图，但不一定能还原真实交易、换月和成本。必须尽量获取所有月份合约。

### 10.2 未来信息泄漏

例如用当天收盘后的持仓量选择当天从开盘开始持有的主力合约，就是未来信息。主力选择、因子和下单时间必须明确错开。

### 10.3 随机拆分训练集和测试集

机器学习阶段不能随机打乱所有日期，应使用时间顺序拆分或滚动训练：

```text
训练期 → 验证期 → 测试期 → 向前滚动
```

### 10.4 忽略换月

策略即使因子值不变，主力合约变化时也会发生平旧仓、开新仓，产生交易成本。

### 10.5 把价格收益直接当成账户盈亏

期货盈亏和价格变化、合约乘数、手数、多空方向、手续费及保证金有关。研究排序可以使用标准化收益，但账户回测要使用合约级盈亏。

### 10.6 对连续价格复权后直接计算真实 P&L

复权连续价格可用于技术指标，但回测成交必须回到实际合约价格。

### 10.7 忽略数据授权

学校数据库数据通常受授权限制，应该：

- 不把原始数据提交 Git；
- 不把数据公开发送给无权限者；
- 不提交账号、Cookie 或密码；
- 报告中注明数据来源；
- 遵守数据库导出和使用限制。

## 十一、第一版框架的验收标准

- 新同伴只看 README 就能安装；
- 没有学校数据也能运行完整演示；
- CSV 和 Excel 均能导入；
- 原始文件不会被程序修改；
- 数据异常会产生明确错误；
- 主力合约选择使用滞后信息；
- 每次换月都有记录；
- 因子计算不会读取未来数据；
- 回测包含手续费、滑点和换月成本；
- 结果可以重复生成；
- 测试全部通过；
- 上财数据、账号和密码不会进入 Git；
- 以后增加模型时，不需要修改数据和回测核心。

## 十二、未来如何增加机器学习和 LLM

### 12.1 机器学习

框架中提前保留统一接口：

```python
model.fit(features, labels)
prediction = model.predict(features)
```

以后无论增加线性回归、Random Forest、LightGBM、XGBoost 或神经网络，其输出都转换为：

```text
trade_date | product | prediction
```

组合和回测层不需要知道模型内部细节。

### 12.2 LLM

LLM 更适合以后用于：

- 阅读研究文献；
- 解释因子表现；
- 自动生成回测摘要；
- 检查实验配置；
- 辅助编写研究日志；
- 从公告或报告中提取结构化信息。

不建议让 LLM 直接负责底层行情清洗、账户盈亏计算、主力换月、风控约束或无校验地生成交易指令。这些环节必须保持确定性和可测试性。

## 十三、推荐实施策略

总体上，最合适的实施策略是：

> 先以“上财平台人工导出文件”为真实数据入口，以“合成数据”为无权限情况下的可运行保障，完成数据标准化、期货换月、基础因子、组合回测和报告，再接入机器学习。

真正落实时，应先建立目录、环境和合成数据演示，不急着处理复杂模型，也不需要一次性获得全部上财数据。
